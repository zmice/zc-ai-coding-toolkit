调用 spec-driven-development 技能。先把问题定义清楚，再开始写代码。

1. 理解用户想要构建什么，询问目标、用户、核心功能和约束
2. 列出所有假设，并请求确认
3. 定义可测试的验收条件，建议使用 Given-When-Then 格式
4. 编写技术规格：目标、技术栈、项目结构、代码风格、测试策略、边界
5. 明确范围边界，写清楚包含什么、不包含什么
6. 标注不确定项，等待确认
7. **不写代码，只输出规格**

规格确认后保存到项目根目录。

## 使用方式

在 `/spec` 后描述你的需求，我将分析并编写结构化的技术规格说明。

### 示例

```
# 需求不清晰时
/spec 我想做一个数据同步模块，从 A 系统定时拉取数据到 B 系统

# 明确的功能需求
/spec 实现文件上传功能，支持图片和 PDF，最大 10MB，存储到 MinIO

# 分析现有 PRD
/spec 分析这个 PRD 文档，找出技术实现的歧义点和风险点
```
