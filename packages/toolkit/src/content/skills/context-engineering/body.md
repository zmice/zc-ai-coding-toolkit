# 上下文工程

## 何时使用

- 开始新的编码会话或新子任务时
- 输出质量下降，开始出现幻觉、忽略约定或跑偏
- 任务切换导致上下文混杂时
- 需要为代理准备最小但足够的输入材料时

## 输入前提

- 已知道当前任务真正需要哪些信息
- 愿意主动裁剪无关上下文
- 接受规则文件、规格、源码、错误输出有不同优先级

## 执行步骤

1. 先装载稳定规则：规则文件、长期约定、关键边界
2. 再装载任务级上下文：规格片段、相关源码、测试、错误输出
3. 控制上下文体积，只保留当前任务需要的信息
4. 任务切换时主动压缩总结，清理旧上下文
5. 发现上下文冲突或缺失时，显式暴露而不是猜测

## 成功标准

- 代理看到的是当前任务需要的最小充分上下文
- 规则、规格、源码和错误信息的层级清晰
- 上下文切换不会把旧问题带入新任务
- 遇到冲突时能清楚指出需要人类决策的点

## 相关原则

- 少而准，比多而杂更有效
- 规格和规则优先于猜测
- 压缩上下文是持续动作，不是最后补救

## 与其他技能的衔接

- 与 `codebase-onboarding` 配合，先建立地图再装载局部上下文
- 与 `context-budget-audit` 配合，检查上下文是否膨胀
- 为几乎所有实现和审查技能提供前置支持
