## 概述
从多个角度评审 Spec/Plan，发现单一视角容易遗漏的问题。

## 对应技能
multi-perspective-review

## 使用方式
`/plan-review [--product] [--engineering] [--design] [--devex]` — 可选择评审视角，默认全部

## 评审流程

### 1. 读取 Spec/Plan
- 读取当前 Spec 或 Plan 文档
- 识别文档类型和核心内容
- 提取关键决策点

### 2. 多视角评审

#### 产品视角（--product）
- 用户价值是否清晰？
- 边界场景是否覆盖？
- 验收标准是否可测？
- 与现有功能是否冲突？

#### 工程视角（--engineering）
- 技术方案是否可行？
- 性能和扩展性考虑是否充分？
- 是否有技术债务风险？
- 依赖和集成点是否明确？

#### 设计视角（--design）
- 用户体验流程是否顺畅？
- 异常状态处理是否完整？
- 与现有设计系统是否一致？
- 可访问性是否考虑？

#### 开发者体验视角（--devex）
- API 设计是否直觉化？
- 文档和示例是否充分？
- 配置复杂度是否合理？
- 错误信息是否有帮助？

### 3. 汇总发现
每个视角独立输出发现和建议，标注风险等级：
```
📋 多视角评审报告

🔴 Critical（阻断）:
  [视角] [问题描述] → [建议]

🟡 Warning（需关注）:
  [视角] [问题描述] → [建议]

🔵 Suggestion（改进建议）:
  [视角] [问题描述] → [建议]
```

### 4. 门控判定
- 存在 Critical 问题 → 阻断，必须修复后才能继续
- 仅 Warning → 可继续，但建议在实现前处理
- 仅 Suggestion → 可继续，记录为后续改进
