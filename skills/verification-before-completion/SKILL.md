# Verification Before Completion

## Overview

声明工作完成而不验证，是不诚实而非高效。

**铁律：证据先于断言，永远如此。**

如果你没有在当前消息中运行验证命令，你就不能声称它通过了。违反这条规则的字面意思，就是违反这条规则的精神。

## When to Use

- 即将声明任何形式的"完成"或"通过"之前
- 提交代码、创建 PR、标记任务完成之前
- 从一个任务切换到下一个任务之前
- 将工作委派给子代理并接收结果之后
- 任何表达满意或积极状态的陈述之前

## 门控函数

在声明任何状态之前，执行以下步骤：

```
1. IDENTIFY — 什么命令能证明这个声明？
2. RUN     — 执行完整的验证命令（新鲜的、完整的）
3. READ    — 完整读取输出，检查退出码，统计失败数
4. VERIFY  — 输出是否支持你要声明的状态？
   - 如果否：报告实际状态 + 证据
   - 如果是：带证据报告声明
5. CLAIM   — 只有此时，才能做出声明

跳过任何一步 = 欺骗，不是验证
```

## 验证证据表

每种声明都有对应的最低验证要求：

| 声明 | 所需证据 | 不充分的证据 |
|------|---------|------------|
| 测试通过 | 测试命令输出：0 failures | 上一次运行结果、"应该能过" |
| Lint 通过 | Linter 输出：0 errors | 部分检查、推测 |
| 构建成功 | 构建命令：exit 0 | Linter 通过、日志看着没问题 |
| Bug 已修复 | 原始症状的测试：通过 | 代码改了，假设修好了 |
| 回归测试有效 | 红绿循环验证完成 | 测试只通过了一次 |
| 子代理完成 | VCS diff 显示变更内容 | 子代理报告"成功" |
| 需求已满足 | 逐项对照清单 | 测试通过 ≠ 需求满足 |

## 红旗信号 — 立即停止

当你发现自己正在做以下事情时，**停下来**：

- 使用"应该"、"大概"、"看起来"等模糊措辞
- 在验证之前表达满意（"太好了！"、"完美！"、"搞定！"）
- 即将提交/推送/创建 PR 但没有运行验证
- 信任子代理的成功报告而不独立验证
- 依赖部分验证就得出整体结论
- 心想"就这一次跳过吧"
- 疲惫时想快速结束工作
- **任何**暗示成功但未运行验证的措辞

## 合理化防御

| 借口 | 现实 |
|------|------|
| "现在应该能用了" | 运行验证命令 |
| "我很有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Linter 通过了" | Linter ≠ 编译器 ≠ 测试 |
| "子代理说成功了" | 独立验证 |
| "我累了" | 疲惫 ≠ 免责 |
| "部分检查够了" | 部分证明不了什么 |
| "换个说法就不算了" | 精神高于字面 |

## 关键验证模式

### Five-Step Verification Gate — 验证五步门禁

每次声明任务完成前，必须按顺序执行以下五步。跳过任何一步都不算完成验证。

#### 五步流程

| 步骤 | 动作 | 要求 |
|------|------|------|
| 1. IDENTIFY | 确定验证命令 | 什么命令可以证明这个声明？列出所有需要运行的命令 |
| 2. RUN | 执行完整命令 | 运行实际命令，不是引用缓存结果，不是回忆之前的输出 |
| 3. READ | 完整读取输出 | 读完整个输出，检查 exit code，计数通过/失败数，不要只看最后一行 |
| 4. VERIFY | 验证输出内容 | 输出确实证实了你的声明吗？不是"看起来对"，是"确实对" |
| 5. CLAIM | 附带证据声明 | 引用具体的输出行数、通过数、exit code 来支撑你的声明 |

#### 禁止清单

| 禁止的表述 | 为什么是错的 | 应该怎么做 |
|-----------|------------|----------|
| "测试应该能通过" | "应该"不是证据 | 运行测试，贴出结果 |
| "看起来没问题" | "看起来"是主观判断 | 运行验证命令，读完输出 |
| "我确信能工作" | 确信 ≠ 证据 | 用命令输出证明 |
| "之前运行过了" | 代码可能已变更 | 重新运行，获取新鲜结果 |
| "Lint 通过了所以没问题" | Lint ≠ 编译 ≠ 测试 ≠ 运行时 | 每个层面单独验证 |
| "只改了一行应该没影响" | 一行代码可以破坏整个系统 | 运行完整测试套件 |

#### 验证层级

完整验证应覆盖所有相关层级，每个层级独立确认：

```
Lint/Format → Compile/Build → Unit Tests → Integration Tests → E2E Tests（如适用）
     ✓              ✓              ✓              ✓                ✓
```

上一层通过不代表下一层通过。每个层级都需要独立运行和确认。

### 测试验证
```
✅ [运行测试命令] → [看到：34/34 pass] → "所有测试通过"
❌ "改完了应该能过" / "看起来对了"
```

### 回归测试验证（TDD 红绿）
```
✅ 写测试 → 运行（通过）→ 撤销修复 → 运行（必须失败）→ 恢复修复 → 运行（通过）
❌ "我写了回归测试"（但没有红绿验证）
```

### 构建验证
```
✅ [运行构建] → [看到：exit 0] → "构建通过"
❌ "Linter 通过了"（Linter 不检查编译）
```

### 需求验证
```
✅ 重读计划 → 创建逐项清单 → 逐条验证 → 报告缺口或完成
❌ "测试通过了，阶段完成"
```

### 子代理委派验证
```
✅ 子代理报告成功 → 检查 VCS diff → 验证变更内容 → 报告实际状态
❌ 信任子代理报告
```

## 与其他技能的衔接

- **test-driven-development** — TDD 的红绿循环本身就是验证模式的一种
- **code-review-and-quality** — 审查清单中的"验证故事"部分由此技能驱动
- **sdd-tdd-workflow** — 每个阶段的门控检查都应遵循验证铁律
- **subagent-driven-development** — 子代理结果必须独立验证，不可直接信任

## 底线

验证没有捷径。

运行命令。读取输出。**然后**声明结果。

这是不可谈判的。
