# Parallel Agent Dispatch

## Overview

当实现计划中有多个独立任务时，通过并行子代理同时执行，大幅加速开发。每个子代理在隔离的上下文中工作，完成后汇总结果并检测冲突。

**核心模式：扇出（Fan-Out）→ 并行执行 → 扇入（Fan-In）**

## When to Use

```
有多个待执行任务？ ─── 否 ──→ 不需要此技能
    │
   是
    ▼
任务之间独立？ ─── 否 ──→ subagent-driven-development（串行）
    │
   是
    ▼
任务 >2 个？ ─── 否 ──→ 串行可能更简单
    │
   是
    ▼
parallel-agent-dispatch ✓
```

适用场景：
- 多个独立模块可以同时实现
- 多个独立测试文件可以同时编写
- 多个不相关的 Bug 可以同时修复
- 代码审查的不同维度可以并行进行

不适用场景：
- 任务之间有依赖关系（Task B 需要 Task A 的输出）
- 任务修改同一个文件
- 任务需要共享状态

## 执行流程

### 阶段 1：分析与分组（Analysis）

```
读取计划中的所有任务
    │
    ▼
构建依赖图
    │
    ▼
识别可并行的任务组
    │
    ▼
检查文件冲突（两个任务是否修改同一文件？）
    │
    ▼
确定并行批次
```

**关键检查：**
- 同一文件不能被两个并行任务修改
- 有依赖关系的任务必须串行
- 共享测试 fixtures 的任务要注意隔离

### 阶段 2：扇出（Fan-Out）

为每个并行任务分派子代理：

```
控制器
    ├─→ 子代理 A：Task 1（文件集 A）
    ├─→ 子代理 B：Task 2（文件集 B）
    └─→ 子代理 C：Task 3（文件集 C）
```

**每个子代理接收：**
- 完整的任务描述
- 允许修改的文件列表（严格限定）
- 项目约定和模式示例
- 测试策略
- **明确说明不要触碰其他子代理的文件**

### 阶段 3：并行执行

每个子代理独立工作：
1. 遵循 TDD 循环（红 → 绿 → 重构）
2. 只修改分配给自己的文件
3. 运行自己范围内的测试
4. 报告状态（DONE / BLOCKED / NEEDS_CONTEXT）

### 阶段 4：扇入（Fan-In）

收集所有子代理的结果：

```
子代理 A 完成 ──┐
子代理 B 完成 ──┼──→ 结果汇总 → 冲突检测 → 集成测试
子代理 C 完成 ──┘
```

**汇总检查清单：**
- [ ] 所有子代理报告完成
- [ ] 无文件冲突（检查 VCS diff）
- [ ] 运行完整测试套件（不仅是各自的测试）
- [ ] 检查跨模块的接口一致性
- [ ] 合并所有变更

## 冲突检测

即使预先分配了文件集，仍需检测以下冲突：

| 冲突类型 | 检测方法 | 解决方式 |
|---------|---------|---------|
| 文件冲突 | 两个子代理修改了同一文件 | 手动合并或串行重做 |
| 接口冲突 | 模块 A 的输出与模块 B 的输入不匹配 | 修复接口层 |
| 命名冲突 | 不同子代理对同一概念使用不同命名 | 统一命名后批量替换 |
| 依赖冲突 | 子代理引入了冲突的依赖版本 | 协调统一版本 |

## 失败处理

**单任务失败不阻塞其他任务。**

```
子代理 A：✅ DONE
子代理 B：❌ BLOCKED
子代理 C：✅ DONE
                │
                ▼
       先合并 A 和 C 的结果
       然后单独处理 B（串行重试或重新分析）
```

处理优先级：
1. 收集所有成功结果
2. 运行集成测试（仅成功部分）
3. 分析失败原因
4. 对失败任务：提供更多上下文 / 用更强模型 / 拆分为更小任务

## 批次策略

大型计划不必一次全部并行：

```
批次 1：Task 1, 2, 3（并行）→ 汇总 → 集成测试
    │
    ▼
批次 2：Task 4, 5（并行，依赖批次1的输出）→ 汇总 → 集成测试
    │
    ▼
批次 3：Task 6（串行，依赖批次2）→ 最终测试
```

**批次划分原则：**
- 同一批次内的任务互相独立
- 下一批次可以依赖上一批次的输出
- 每批次后运行集成测试
- 批次大小根据可用资源调整

## 与其他技能的衔接

- **subagent-driven-development** — 姊妹技能：串行版本，适合有依赖的任务链
- **planning-and-task-breakdown** — 上游：生成带依赖关系标注的任务计划
- **verification-before-completion** — 扇入阶段的集成测试遵循验证铁律
- **code-review-and-quality** — 所有并行结果合并后进行统一审查

## Red Flags

- 并行执行有依赖关系的任务
- 两个子代理修改同一文件
- 跳过扇入阶段的集成测试
- 不检查跨模块接口一致性
- 子代理间共享上下文（违背隔离原则）
- 单个子代理失败就放弃所有结果
- 不运行完整测试套件就声明并行执行完成
