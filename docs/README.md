# Docs Responsibilities

`docs/` 只保留长期有价值的仓库文档，不作为过程性计划堆放区。

## 保留什么

- `docs/adr/`
  - 架构决策记录
  - 记录为什么这样设计，而不是实现步骤
- `docs/architecture/`
  - 当前仍然有效的长期规则、边界和项目地图
  - 例如分层、命名、上游治理、发布策略、中文化规则
- `docs/release-*.md`
  - 仍会在后续发布中反复使用的 checklist 和 guide
- `docs/usage-guide.md`
  - 面向使用者的长期操作说明，包括 CLI 安装、更新和平台内容安装

## 不保留什么

- 一次性实施计划
- 已完成任务的阶段拆解
- 临时 spec 草稿
- 仅服务某次重构冲刺、完成后不再复用的过程文档

这类内容应当：

- 在实现完成后删除
- 或仅保留在 git history 中

## 维护规则

- 新增文档前先问：这份文档 3 个月后是否仍值得阅读？
- 如果答案是否定的，不要放进 `docs/`
- 如果文档只是任务推进用，请保留在会话、提交历史或 issue/PR 中，而不是仓库长期文件
